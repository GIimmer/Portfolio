User Dashboard

Overview

User Dashboard is a web application amalgamating many common operations together in one package; these include secure sign in with validations, CRUD, Models and SQL relationships replicating a classic Facebook-type application (with some interesting extras thrown in). Note that this will soon be replaced with a more complex eCommerce-styled application.

The image above is the front page from which we begin. Let's walk through my application in the shoes of one Morgan Freeman; a true honor. Note that I will frequently clip out the linking navbar at the top - though it is always there for UX - for the improvement of this pages length and readability.

Wanting to check on what all of his friends are doing Morgan Freeman loads up the new User Dash application. Not having an account he clicks 'start' which takes him to the sign in page.

Being overly excited; Morgan fills out most of his information correctly, but accidentally types in the single characters 'M' and 'F' for his first and last names. Smartly, the website alerts him to this, and he quickly rectifies the issue.



Notice that following basic validations including simple length and regex parameters for the various input fields, passwords are immediately hashed using BCrypt before ever being stored in the database.

Having successfully signed in Mr. Freeman is overjoyed to see all of the friends upon who's walls he can now comment! However he notices that two users; Top Turtle and Riley, have higher user privileges than himself. Disturbed he clicks on 'profile' at the top of his screen to correct this issue.

Morgan is pleased to see that his information has been loaded into the input fields prior to his own interaction. However being a standard user he is unable to adjust his own user privileges, he leaves a curt note on Riley's wall.



These code snippets primarily show the use of session to retrieve data about the current user and react appropriately. Changes in the view include 'All Users' becoming 'Manage Users', while an additional column allowing the admin to edit and delete users is added. In the code above you may have noticed that the very first user to log in is automatically granted level 9. Finally the 'Delete message' option is only granted to the original creator of said message (and there is backend code to ensure users can't just mischievously type in the corresponding Delete URL GET request).

Quick note - Directly to the right you can see an example of "The Wall"; each user has one. Other users can post messages and replies on these walls. I'm using Slow Rabbits', as an example (for reason of my wall being extraordinarily long with test messages). Nonetheless it looks much the same.


While browsing my wall , I notice a new message. Morgan seems steamed up - I immediately become fearful - and after posting a quick reply I select 'edit' in the admin 'actions' column. Being an admin I have access to editing every single user, with one more option than they have - 'User Level'.

Feeling apologetic and short of breath, I change his admin privileges to match my own. At the bottom right you will notice that I have elevated Morgan, as is his birthright.
